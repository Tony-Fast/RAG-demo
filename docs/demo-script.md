# RAG 知识库系统 - 演示脚本

本文档提供完整的面试演示流程和脚本，帮助您有效地展示 RAG 知识库系统的功能和价值。

## 演示概览

演示总时长：**10-15 分钟**

### 演示流程

1. **开场介绍**（2 分钟）- 系统概述和技术选型
2. **功能实操**（5 分钟）- 完整功能演示
3. **技术深度**（3 分钟）- 核心算法讲解
4. **Q&A 准备**（5 分钟）- 常见问题回答

---

## 第一部分：开场介绍

### 开场白（30 秒）

> "大家好，今天我要向大家演示的是一个基于 RAG（检索增强生成）技术的智能知识库问答系统。这个项目展示了我对现代 AI 应用开发的理解，包括向量数据库、大语言模型集成以及完整的前后端架构设计。"

### 技术架构概述（1 分 30 秒）

**后端技术栈：**
- **FastAPI** - 高性能 Python Web 框架，提供异步支持
- **FAISS** - Facebook AI 开发的向量相似性搜索库
- **Sentence-Transformers** - 文本向量化模型
- **DeepSeek API** - 大语言模型推理服务

**前端技术栈：**
- **React 18 + TypeScript** - 现代前端框架
- **Tailwind CSS** - 原子化 CSS 框架
- **Shadcn/UI** - 专业的 UI 组件库

**为什么这样选择？**
1. **FastAPI**：性能优秀，异步支持好，自动化 API 文档
2. **FAISS**：Facebook 开源的向量数据库，支持高效的相似性搜索
3. **DeepSeek**：国产优秀的大模型，性价比高
4. **前端**：展示现代前端开发能力，TypeScript 提供类型安全

## 第二部分：功能实操

### 场景一：上传文档（1 分钟）

**操作步骤：**
1. 打开浏览器，访问应用地址
2. 点击"文档"标签页
3. 拖拽上传一个测试文档（PDF 或 Word）

**演示话术：**
> "首先，我来演示文档上传功能。系统支持多种格式，包括 PDF、Word、Excel、TXT 等。拖拽完成后，系统会自动处理文档，提取文本内容。"

### 场景二：智能问答（2 分钟）

**操作步骤：**
1. 切换到"问答"标签
2. 输入一个基于上传文档的问题
3. 观察系统如何检索相关片段并生成回答

**演示话术：**
> "现在我来问一个关于刚才上传文档的问题。系统会：
> 1. 将用户问题转换为向量
> 2. 在向量数据库中检索最相似的文档片段
> 3. 将检索结果和原始问题一起发送给大语言模型
> 4. 生成最终的回答，并标注信息来源"

**示例问题：**
如果上传了技术文档，可以问：
- "什么是 RAG 技术？"
- "请总结本文档的核心内容"
- "文档中提到的关键技术点有哪些？"

### 场景三：查看检索来源（1 分钟）

**操作步骤：**
1. 在问答结果中找到"信息来源"部分
2. 点击展开查看详细的引用片段
3. 展示相似度分数

**演示话术：**
> "大家可以看到，每个回答都标注了信息来源。我可以点击展开查看具体的引用片段和相似度分数。这展示了系统的透明度，让用户知道答案来自哪里。"

### 场景四：配置调整（1 分钟）

**操作步骤：**
1. 点击"配置"标签
2. 调整"检索数量（Top-K）"参数
3. 观察参数变化对回答的影响
4. 调整"温度"参数

**演示话术：**
> "系统支持实时配置调整。我可以修改检索数量——这决定了每次查询返回多少个相关片段。也可以调整温度参数，控制回答的创造性。这些配置可以实时生效，无需重启服务。"

## 第三部分：技术深度

### 3.1 向量检索原理（1 分钟）

**演示话术：**
> "RAG 的核心是向量检索。当文档上传时，系统会将文本切分成小块，然后使用 Sentence-Transformers 模型将每段文本转换为高维向量。这些向量存储在 FAISS 数据库中。
> 
> 检索时，用户问题也会被转换为向量，然后通过余弦相似度计算，找出最相似的文档片段。这种语义检索比传统的关键词匹配更加智能。"

**可视化建议：**
- 展示向量空间的概念图
- 解释相似度计算公式

### 3.2 提示词工程（1 分钟）

**演示话术：**
> "为了让大语言模型更好地回答问题，我们设计了专门的提示词模板。提示词包含三个部分：
> 1. **系统提示**：定义 AI 的角色和行为
> 2. **上下文信息**：检索到的相关文档片段
> 3. **用户问题**：原始问题
> 
> 提示词的设计直接影响回答质量，这是一个需要不断优化的过程。"

### 3.3 系统架构设计（1 分钟）

**演示话术：**
> "从架构角度看，系统分为三层：
> 1. **表现层**：React 前端，提供用户界面
> 2. **业务层**：FastAPI 后端，处理文档、检索和生成逻辑
> 3. **数据层**：FAISS 向量数据库和文件系统
> 
> 这种分层架构的优势是职责清晰、易于维护和扩展。"

## 第四部分：常见问题准备

### Q1：为什么选择 FAISS 而不是其他向量数据库？

**建议回答：**
> "FAISS 是 Facebook 开源的向量检索库，有以下优势：
> 1. **高性能**：支持十亿级向量的快速检索
> 2. **易部署**：可以本地部署，无需外部服务
> 3. **灵活性**：支持多种索引类型
> 4. **成熟稳定**：经过大规模生产环境验证
> 
> 对于演示和中小规模应用，FAISS 是很好的选择。如果需要云服务，也可以轻松切换到 Pinecone。"

### Q2：如何处理长文档？

**建议回答：**
> "长文档处理涉及几个关键步骤：
> 1. **文本分块**：将长文档切分成小片段
> 2. **重叠窗口**：相邻块之间有重叠，确保信息不丢失
> 3. **元数据保留**：记录每个块的来源信息
> 
> 当前系统默认使用 1000 字符的分块大小和 200 字符的重叠，这个参数可以通过配置界面调整。"

### Q3：如何优化检索质量？

**建议回答：**
> "检索质量优化可以从几个方面入手：
> 1. **调整 Top-K 参数**：增加返回的片段数量
> 2. **设置相似度阈值**：过滤低相似度的结果
> 3. **重排序**：对初步检索结果进行二次排序
> 4. **混合检索**：结合关键词检索和向量检索
> 
> 在实际应用中，需要根据具体场景反复测试和调整。"

### Q4：系统有什么局限性？

**建议回答：**
> "当前的演示版本有几个局限：
> 1. **单用户设计**：没有用户认证和隔离
> 2. **同步处理**：文档处理是同步的，大文件会阻塞
> 3. **本地存储**：数据持久化依赖本地文件系统
> 
> 生产环境可以考虑添加异步队列、用户系统、云存储等组件。"

### Q5：为什么选择 DeepSeek 而不是其他大模型？

**建议回答：**
> "选择 DeepSeek 有几个考虑：
> 1. **性价比高**：相比 GPT-4，API 成本更低
> 2. **中文能力强**：对中文理解出色
> 3. **响应速度快**：延迟较低
> 4. **国产化**：符合技术自主可控的趋势
> 
> 系统设计是解耦的，可以轻松切换到其他大模型 API。"
